Question 2 - Algorithm (20pts)

The first step in development is to write an algorithm that computes the set of 3-digit numbers that are equal to the sum of the cubes of the digits. As a hint, there are 4 such numbers. There are a couple of different ways of structuring the loop(s) for this task. Feel free to use the approach that feels most natural to you. The two general approaches can broadly be described as:

  • Do I take each of the 900 numbers, dissect it, and test to see if it fits the criteria?
  • Do I construct each of the 900 numbers from their digits and test to see if they fit?

Having written this both ways, I'm of the opinion that the two approaches are equally easy/difficult. Don't run to Google to find a "solution." Your brain is a much more reliable resource. If you're not to that point yet, remember that one of the purposes of college is to help you get to where you have more confidence in your own brain than you do in what others tell you.

Which approach did you take? Why did you take that approach? Write out the algorithm in more detail in English. Think about every step you have to take.


Question 3 - Starting the Implementation (10pts)

Now that you've written the algorithm, your next step is to design how your data will be organized. For each of the data items described in your algorithm list its name and a short description of its role.


Question 4 - Write the code (15pts)

You now have an algorithm and your data structures for this problem. These two items together are what make a program. So it's time to translate what you've done into Tranquility. Write your program in a file called cubes.t and compile it with the tranqc command on tux. Fix any problems that the compiler reports. What errors did the compiler report? How did you fix them?


Question 5 - Testing your program (15pts)

Getting a program to compile without errors isn't the endpoint of programming; it's just the beginning. Create a file called cubes.html that loads the machine language code created by the Tranquility compiler. Follow the examples in class and the "Tranquility Programmer's Manual." This is where the real work of debugging begins. Any number of possible errors can arise. It's at this point that it's critical to be able to trace what the program should do and verify that it is working correctly.

After you think you've got it working, try someone else's implementation and see if they got the same numbers you did. Whose did you try, and did they get the same results?
